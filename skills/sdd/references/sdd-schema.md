# SDD Schema Reference

Format reference for schema artifacts and the schema conformance lifecycle.

## 1. Schema Evidence Annotation

Used inline in spec scenarios (baseline and delta) when a requirement maps directly to a schema path.

```markdown
#### Scenario: {ScenarioName}

- **GIVEN** {precondition}
- **WHEN** {action}
- **THEN** {expected outcome}

**Schema reference:** {schema-type} `{path/to/element}` — {what to look for}
```

Examples:

```markdown
**Schema reference:** openapi.yaml `POST /users` request body → `email: string(format=email)`, `name: string(minLength=1)`
**Schema reference:** db-schema.sql `users` table → `email VARCHAR NOT NULL UNIQUE`
**Schema reference:** schema.graphql `Mutation.createUser` → `input: CreateUserInput`
**Schema reference:** users.proto `CreateUserRequest` → `email string`, `name string`
```

Rules:

- Only add when a direct schema path can be cited — omit if speculative
- `Schema reference` is bold, followed by the schema filename and path
- Generated by `sdd-derive`; may be added manually during `sdd-propose`

## 2. Schema Expected Changes Format

Location: `.specs/changes/<name>/schemas/expected.md`

```markdown
# Expected Schema Changes: {Change Name}

## {Schema Type} (e.g., OpenAPI, Database DDL, GraphQL SDL)

### Added

- {operation or field}: {brief description}

### Modified

- {operation or field}: {old behavior} → {new behavior}

### Removed

- {operation or field}: {reason}
```

Rules:

- Written at proposal time before implementation
- Used by `sdd-verify` to cross-check the actual schema diff
- Prose descriptions only — no schema syntax required
- Omit empty sections (Added/Modified/Removed)

## 3. Schema Config Format

Location: `.sdd/schema-config.yaml`

```yaml
# .sdd/schema-config.yaml
# Defines how SDD extracts and tracks schema artifacts for this project.
# All fields except 'schemas' are optional.
#
# Schema types shown below (openapi, graphql-sdl, sql-ddl, proto) are examples only.
# Any schema-like artifact can be configured — use any descriptive type value for
# formats not listed here (e.g., json-schema, avro, custom).

schemas:
  openapi:
    type: openapi
    # Optional: authored/committed schema already in the repo (aspirational/design intent)
    authored: docs/openapi.yaml
    # Optional: command to auto-generate a schema snapshot from the running codebase
    generate:
      command: curl -s http://localhost:8000/openapi.json
      # Other examples (not exhaustive — any framework that can emit a schema works):
      # FastAPI (Python):       python -c 'from app.main import app; import json; print(json.dumps(app.openapi()))'
      # NestJS (TypeScript):    npx ts-node -e 'require("./src/swagger").printSchema()'
      # Spring Boot (Java):     curl http://localhost:8080/v3/api-docs
      # Rails API (Ruby):       bundle exec rake rswag:specs:swaggerize
      # Go (echo/gin):          go run ./cmd/schema/main.go
      # Laravel (PHP):          php artisan l5-swagger:generate

  graphql:
    type: graphql-sdl
    authored: schema.graphql
    generate:
      command: npx get-graphql-schema http://localhost:4000/graphql
      # Or for schema-first projects, just reference the canonical SDL file directly

  db:
    type: sql-ddl
    authored: migrations/schema.sql
    generate:
      command: pg_dump --schema-only --no-owner $DATABASE_URL
      # Other examples: prisma db pull, mysqldump --no-data, sqlite3 app.db .schema

  protobuf:
    type: proto
    # Proto files are typically declarative — just snapshot them directly
    authored: proto/
    generate:
      command: find proto/ -name '*.proto' | sort | xargs cat
```

Rules:

- `type` is informational — used to select the appropriate diff strategy; values shown are examples, any descriptive string is valid
- `authored` path is relative to the project root; omit if no authored schema exists
- `generate.command` must be runnable from the project root; omit if extraction is not possible
- Each schema entry is independent — configure only what the project has
- Store this file in `.sdd/`, not in `.specs/` (it's tooling config, not a spec)

## 4. Schema Lifecycle Policy

Each SDD skill plays a distinct role in the schema lifecycle — act only within your role.

| Skill           | Schema role                                                                                |
| --------------- | ------------------------------------------------------------------------------------------ |
| `sdd-derive`    | Detect artifacts; generate baseline snapshots; annotate specs with `**Schema reference:**` |
| `sdd-translate` | Generate bootstrap snapshots; initialize `.specs/schemas/` for a new SDD adoption          |
| `sdd-propose`   | Capture before-snapshot; write expected changes in `schemas/expected.md`                   |
| `sdd-verify`    | Generate after-snapshot; cross-check diff against delta spec requirements                  |
| `sdd-sync`      | Promote verified after-snapshots into the shared `.specs/schemas/` baseline                |

### Missing-Config Policy

Apply consistently across all SDD skills:

| Condition                                                    | Action                                                                             |
| ------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| `schema-config.yaml` exists                                  | Run extraction; proceed with schema operations                                     |
| No config; schema artifacts present; not yet suggested       | One-time suggestion via `suggested-tools` pattern; skip silently after first offer |
| No config; no artifacts detected, or suggestion already made | Skip silently                                                                      |

Skills that skip silently should note the omission in their output where relevant (e.g., `sdd-verify` notes that conformance was not checked).
